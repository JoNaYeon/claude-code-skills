# Prototype Goal

이번 프로토타입이 증명해야 할 핵심 가설을 한 문단으로 기술합니다. 프로토타입은 "이게 되는가?"를 확인하는 것이지 완성품을 만드는 것이 아닙니다.

<!-- 예: "HR 관리자가 대시보드에서 팀별 휴가 현황을 30초 이내에 파악할 수 있는지 검증한다. 특히 캘린더 뷰가 직관적인지, 필터가 기대대로 동작하는지가 핵심이다." -->

## What Must Be Visible

프로토타입에서 반드시 동작하거나 보여야 하는 것입니다. 이것이 없으면 검증 불가.

- <!-- 예: 팀원 5명 이상의 휴가 데이터가 캘린더에 표시 -->
- <!-- 예: 부서 필터 클릭 시 실제로 필터링 동작 -->
- <!-- 예: 한 직원 클릭 시 연차 잔여일이 표시 -->

## Flow To Demonstrate

프로토타입 데모 시 보여줄 사용자 플로우입니다. 순서대로 시연합니다.

1. <!-- 예: 대시보드 접속 → 이번 달 전체 팀 캘린더 뷰 확인 -->
2. <!-- 예: "개발팀" 필터 선택 → 개발팀만 표시되는지 확인 -->
3. <!-- 예: "홍길동" 클릭 → 상세 패널에서 연차 잔여일 확인 -->
4. <!-- 예: 날짜 범위를 다음 달로 변경 → 예정 휴가 확인 -->

## Open Questions To Test

이 프로토타입으로 답을 얻고 싶은 질문입니다.

- <!-- 예: 캘린더 뷰 vs 테이블 뷰 — 어느 쪽이 더 직관적인가? -->
- <!-- 예: 직원 이름을 몇 글자까지 마스킹해야 식별과 개인정보 보호가 양립하는가? -->
- <!-- 예: 한 화면에 최대 몇 명까지 표시해야 스크롤 없이 사용할 수 있는가? -->

## What Can Stay Rough

프로토타입에서 완성도가 낮아도 되는 부분입니다. 여기에 시간을 쓰지 않습니다.

- <!-- 예: 디자인 — 색상/폰트는 기본값 사용 가능 -->
- <!-- 예: 데이터 — 하드코딩된 더미 데이터로 충분 -->
- <!-- 예: 성능 — 5명 데이터로만 테스트, 대량 데이터는 무시 -->
- <!-- 예: 인증 — 로그인 없이 바로 접근 가능해도 됨 -->
- <!-- 예: 에러 처리 — happy path만 구현 -->

## Review Checklist

프로토타입 완성 후 리뷰 전에 확인할 항목입니다.

- [ ] "What Must Be Visible" 항목이 모두 동작하는가?
- [ ] "Flow To Demonstrate" 순서대로 시연 가능한가?
- [ ] 데모 중 크래시나 블로킹 에러가 없는가?
- [ ] 리뷰어(HR 관리자/PM)에게 사전 안내할 사항을 정리했는가?
- [ ] 프로토타입의 한계(What Can Stay Rough)를 리뷰어에게 설명할 준비가 되었는가?
